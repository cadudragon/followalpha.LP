# FollowAlpha.LP — Implementation Plan (phased delivery with gates)

Authored 2026-06-14. Phases are strictly ordered. Each phase has a **Crivo (gate)** that must be passed before the next phase starts — no exceptions, no "I'll fix it later." Each phase ends with a commit tagged `phase-N-done` only after its gate passes.

## Governing principle: prove value before building UI

The frontend is **not** built until the tool has demonstrated, on real watchlist pools and the principal's real context, that it surfaces true, non-obvious, decision-relevant LP information. There is an explicit **GO/NO-GO Value Validation gate (Phase 5)** between the headless analytical product and any UI work. If the Range Advisor cannot help choose pools/ranges better than eyeballing a DEX screen, we do not paint a screen on top of it — we rethink the fundamentals. A pretty UI over a worthless engine is the most expensive failure mode; this plan makes it structurally impossible.

Product-order decision (2026-06-15): the first value proof is **Range Advisor / Range Intelligence with descriptive replay**, not LP-Audit. LP-Audit remains valuable, but moves later as calibration and self-confrontation against the principal's history.

## How the Crivo works (roles)

Two kinds of gate, mirroring the research-program discipline:

- **Mechanical gates** (Phases 0-4, 6-7): evidence is automated — CI green, tests, determinism, reproducibility. The implementing agent self-reports with evidence; CI is the impartial judge. Pass is binary.
- **Judgment gates** (Phase 5 value validation; Phase 8 forward-tracking review): the implementing agent produces an evidence dossier; the **analyst adjudicates** the evidence skeptically; the **principal decides** GO/NO-GO. This is the same separation used to kill the research programs — the implementer computes, the analyst judges, the principal owns the call.

An agent that reaches a gate writes a short **Gate Report** (what was built, evidence, known gaps) and stops. It does not self-certify a judgment gate.

Persistent deferrals and modelling choices that must survive context resets live in
`docs/OPEN-DECISIONS.md`. Commit bodies may summarize them, but they are not the durable source of
truth.

**Deployment-timing decision (2026-06-17):** the Collector is *designed* always-on (tick-liquidity distributions cannot be reconstructed retroactively), but standing up the **always-on Oracle/VPS deployment is deferred until after Phase 3 full has proven sufficient value/edge** — we do not pay for 24/7 infrastructure before the engine has earned it. Until then, **local/intermittent runs are acceptable** for smoke and initial collection, accepting the known loss of tick-liquidity during downtime (never synthetically backfilled). This **does not block the start of Phase 3** (`phase-3-done`'s precondition is `phase-2-done`, the agent gate). The principal's always-on deploy is `CHECKLIST.md` item 2.6, an operational follow-up logged separately when actually performed. Agents never receive VPS credentials.

## Phase 3 — Range Advisor & descriptive replay (first real answer)

This is the first useful product surface, headless. API host (`API-CONTRACT.md`) + CLI for the asset-first decision flow:
- `ClassifyVolRegime` (Module 1, `/regime`) — RANGE/TRENDING/TRANSITION, no direction.
- Pool comparison for an asset — fee tier, volume/TVL, pool IV, competing liquidity.
- `SuggestRangeCandidates` (`/ranges/candidates`) — deterministic candidate bands from a predeclared grid, ranked by IV-vs-RV, band survival, expected fees, IL, and intent fit; **not** an optimizer and not tuned against historical outcomes.
- `EstimateRangeApr` (`/ranges/estimate-apr`) — honest APR: self-dilution, while-in-range vs time-adjusted, 7d/30d sensitivity; QA cross-checked against Metrix Finance.
- `EvaluateRange` (Module 2, `/ranges/evaluate`) — fee APR vs IL + IV vs forecast RV + band survival → `OPEN`/`DONT_OPEN`; every call appends an immutable `DecisionLogEntry` (inputs + hash), even when not opened (RN-03).
- Replay (UC-09 category A, `/ranges/backtest`) — band survival, IV-vs-RV outcomes, fee-APR reconciliation; deterministic, **no optimizer / no threshold tuning** (RN-14).

**Crivo:** regime + pool comparison + verdict + replay reachable via API/CLI on real watchlist pools; decision-log entries immutable and retrievable; outputs deterministic and reproducible; replay proven free of parameter search (code review + test); `422` returned on thin data, never a guess; OpenAPI spec committed and matching `API-CONTRACT.md`. Tag `phase-3-done`. **No frontend code exists yet — by design.**

Tag `phase-4-done`.

## Phase 5 — VALUE VALIDATION GATE (GO/NO-GO — the crivo before any UI)

No new features. Run the complete headless toolset on real watchlist pools and the principal's **real** history, and assemble an **Edge Evidence Dossier**:

1. **Range Advisor usefulness:** on real watchlist assets/pools, does the tool surface pool/range choices, IV-vs-RV context, liquidity crowding, and band-survival facts that the principal would not reliably see by eye?
2. **Mechanism validation (category A):** do band-survival and IV-vs-RV, measured on real pools, **discriminate** good LP conditions from bad ones in a way a human could not eyeball? Show the distributions.
3. **Estimate fidelity:** does `EstimateRangeApr` reconcile acceptably against realized/simulated fees and against Metrix? Document divergences.
4. **Audit retrospective:** does LP-Audit reveal something true and non-obvious — which positions/pools actually had edge, where the displayed APR fooled the principal, fees-vs-IL reality vs perception?
5. **Verdict sanity (honest framing):** on a handful of real past situations, does the `OPEN`/`DONT_OPEN` logic produce defensible calls? This is **descriptive sanity, not proof of edge** — verdict edge is only provable forward (Phase 8), and saying otherwise would be the in-sample self-deception RN-14 forbids.

**Crivo (judgment gate):** the analyst adjudicates the dossier against the discipline (blind-analysis honesty, no overfit claims); the **principal decides GO or NO-GO** to the question: *"Does this tool tell me something true and useful about my LP decisions that I could not see by eye?"*
- **GO** → Phase 6 (frontend) is authorized.
- **NO-GO** → stop. Do not build UI. Revisit the fundamentals (which inputs, which thresholds, what data is missing). The tool earns its frontend; it is not granted one.

Tag `phase-5-go` or record `phase-5-nogo` with the reasons. This gate exists so we never gold-plate a UI onto an engine that has not proven its worth.

## Phase 6 — Frontend (only if Phase 5 = GO)

Next.js per `TECH-STACK.md`/FSD: asset-first flow (Watchlist/Asset View → pools → range evaluator → verdict → decision log), audit dashboard, channel simulator, post-OPEN monitor (UC-08). Consumes the generated OpenAPI client; no business logic client-side; charts via Lightweight Charts/Recharts.

**Crivo:** principal completes the full asset-first flow end-to-end in the browser; verdict screen always shows the inputs behind it; channel results always show the full series including breakouts; post-OPEN monitor shows the fees-vs-IL race per open position. Tag `phase-6-done`.

## Phase 7 — Alerts & verdict-premise drift

`EvaluateAlertRules` in the Collector + `INotificationChannel` (mechanism chosen with the principal); post-OPEN drift flags (regime flip, pool IV below forecast RV) per UC-07/UC-08.

**Crivo:** a real alert fires end-to-end to the configured channel; drift flags show before/after numbers; alerts inform only — no execution path exists. Tag `phase-7-done`.

## Phase 8 — Forward-tracking (the only honest proof of verdict edge) — ongoing

The decision log audits itself on **new** data: verdicts recorded in normal use are compared, over time, against what actually happened. This is category B done correctly — out-of-sample, never in-sample (blind analysis, LP-KNOWLEDGE §6.1).

**Crivo (recurring judgment gate):** at a pre-registered interval, the analyst reviews logged verdicts vs outcomes and reports whether the tool shows real edge. No threshold is changed on the basis of in-sample fitting; any rule change is itself pre-registered. This is what eventually answers "is the edge real" — honestly, with new data, the way the research programs should have from the start.
