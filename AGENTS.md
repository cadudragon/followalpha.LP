# AGENTS.md — FollowAlpha.LP

Instructions for any coding agent (Codex or other) working in this repository. The human principal is the product owner; the quant analyst (architect) authored the contracts in `docs/`. You implement mechanically against them.

## Read first, in this order

1. `LP-KNOWLEDGE.md` — domain knowledge: what this product is, the economics of concentrated liquidity, intent accounting, the 4 modules. **Do not implement anything that contradicts it.**
2. `docs/ARCHITECTURE.md` — the architecture contract: solution layout, dependency rules, layer responsibilities.
3. `docs/IMPLEMENTATION-PLAN.md` — phases with definition of done. Work strictly in phase order.
4. `docs/FSD_FollowAlphaLP_v1.1.md` — approved functional spec: use cases, business rules (RN-01..13), screens. Authoritative for API/frontend behavior.
5. `docs/TECH-STACK.md` — pinned libraries, data sources and chart stack for the analysis layer. Do not introduce dependencies outside it without updating it first.

## Hard rules

1. **Dependency direction is law.** `Domain` references nothing. `Application` references only `Domain`. `Infrastructure` references `Application` + `Domain`. Hosts (`Api`, `Cli`, `Collector`) reference `Application`, and `Infrastructure` only in the DI composition root. Architecture tests enforce this; if they fail, fix your code, never the tests.
2. **Domain stays pure.** No I/O, no `DateTime.Now`/`DateTime.UtcNow` (use `IClock` from Application via parameters), no `Random`, no logging, no async. Deterministic functions and immutable types only.
3. **Golden tests are immutable.** The expected values in `tests/FollowAlpha.LP.Domain.Tests/Golden/` are generated by running the Elsts Python reference (`uniswap-v3-liquidity-math`) as an **oracle** via the script in `tools/oracle/`. Never alter expected values or tolerances to make a test pass — the C# kernel converges to the oracle, never the reverse. **Product code never shells out to Python**; the Python app is oracle + manual exploration only (principal's decision, 2026-06-12).
4. **This tool never executes transactions.** Read-only on-chain access. No private keys, no signing code, no wallet-connect server side. If a task seems to require signing, stop — you misread it.
5. **The decision log is append-only.** Verdicts (`OPEN`/`DONT_OPEN` and channel events) are inserts; never update or delete. Schema changes require an explicit migration with a note in the PR description.
6. **Collected on-chain data are immutable facts.** Snapshot and event tables are append-only; re-ingestion must be idempotent (natural keys / upserts that cannot change history).
7. **No secrets in the repo.** The Graph gateway API key, RPC URLs with keys: environment variables / user-secrets only. `.env*` is gitignored.
8. **Single user today, SaaS-shaped tomorrow.** Every persisted aggregate carries a `TenantId` (constant default tenant for now). No other multi-tenant infrastructure — and no design that would prevent it.
9. **Quality gates**: `nullable` enabled, warnings as errors, all tests green, architecture tests green — before every commit. Run: `dotnet build && dotnet test`.
10. **Don't gold-plate.** Implement the current phase's definition of done; resist adding speculative abstractions not named in the contracts. Extensibility lives exactly where `docs/ARCHITECTURE.md` puts the ports, nowhere else.

## Conventions

- .NET 10, C# latest, file-scoped namespaces, `var` where type is obvious.
- Value objects as `readonly record struct` / `record` as appropriate; entities with private setters.
- Tests: xUnit + FluentAssertions; mocks only at ports (NSubstitute or Moq, pick one and stay consistent).
- Money/price math: `decimal` for human-scale values; `System.Numerics.BigInteger` for raw on-chain integers (liquidity L, Q64.96 sqrtPriceX96). Conversions live in one place (`Domain/Primitives`).
- Commit messages: imperative, scoped (`domain: add tick<->sqrtPrice conversion with golden tests`).
- Frontend (`frontend/`) is a separate Next.js app; it talks to the API via the generated OpenAPI client only. No business logic in the frontend — verdicts come from the API.

## When blocked

If a contract is ambiguous on a **product/research** question (e.g., a formula's benchmark, an intent rule), stop and ask the principal — do not decide. If it's a **mechanical** question (naming, folder, library minor version), resolve it in the spirit of the contracts and note the decision in the PR description.
